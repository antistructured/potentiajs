## Safe data currently exposed by SigilJS

Installed `@weipertda/sigiljs@0.18.0` declares and implements `SigilValidationError` with:

- `code: 'SIGIL_VALIDATION_FAILED'`
- `message`
- `path`
- `expected`
- `actual`
- `toJSON()` returning the same safe fields

The source comment documents `path` as the key path to the failing field and `expected` / `actual` as type summaries.

Current Potentia runtime intentionally collapses all SigilJS validation failures to:

```js
{ message: 'SigilJS contract rejected value' }
```

## Decision

Future implementation should extract structured SigilJS issue data when the thrown value is safely recognized as a SigilJS validation error.

Recognition should be conservative:

- object value
- `code === 'SIGIL_VALIDATION_FAILED'`
- `path` is an array, or absent/invalid and treated as `[]`
- `expected` is a string or safely stringifiable type summary, otherwise `null`
- `actual` is a type summary, not a raw received value; if uncertain, use `null`

## Normalization

A recognized SigilJS failure should normalize to the canonical issue shape:

```js
{
  code: normalizedCode,
  message: safeMessage,
  path: normalizedPath,
  field: fieldFromPath(normalizedPath),
  boundary: boundary,
  source: 'sigil',
  expected: expectedOrNull,
  received: receivedOrNull,
  meta: null
}
```

## Code mapping

Initial mapping should be conservative:

- missing/undefined received with expected present → `missing_required`
- expected/received mismatch → `invalid_type`
- otherwise → `contract_failed`

If SigilJS later exposes richer machine codes, Potentia can map a safe subset.

## Message policy

Do not pass through arbitrary SigilJS message strings by default until every message format is audited.

Initial safe messages:

- `Invalid value`
- `Required value is missing`
- `Contract rejected value`

Human-oriented messages can be improved later, but should be generated by Potentia from safe structured data rather than copied from arbitrary error text.

## Path policy

SigilJS `path` should be normalized as:

- array of string/number segments preserved
- invalid path → `[]`
- string path should not be split unless SigilJS formally documents such shape
- dangerous key names may appear as path segments only after the input boundary has already accepted them; do not use paths to rehydrate data

`field` is derived from normalized `path`.

## Nested fields

Nested object paths are preserved as arrays:

```js
['profile', 'name']
```

and displayed as:

```js
'profile.name'
```

Projection metadata from `describe()` can help tooling know possible fields, but runtime issues must come from validation failure data, not projection guesses.

## Arrays and repeated fields

Array indices should be numeric path segments when SigilJS provides them:

```js
['tags', 1]
```

Display field:

```js
'tags[1]'
```

URL-encoded repeated fields currently parse to arrays of strings. If a SigilJS input contract rejects a repeated field, the issue should reflect the path SigilJS provides. If SigilJS only reports the root, Potentia must not invent an index.

## Unknown SigilJS shapes

If a thrown SigilJS-looking value lacks reliable structured fields, fall back to one root-level issue:

```js
{
  code: 'contract_failed',
  message: 'Contract rejected value',
  path: [],
  field: null,
  boundary: boundary,
  source: 'sigil',
  expected: null,
  received: null,
  meta: null
}
```

## Rejected approach

Do not parse arbitrary human error strings to infer paths, expected types, or received types. String parsing is brittle and can expose unsafe text.

## Deferred

- implementation
- exhaustive SigilJS failure-shape tests
- multi-issue extraction if SigilJS later exposes arrays of issues
- localized messages
- form label mapping

## Blockers

- Current SigilJS appears to throw one validation error at a time; multi-field collection is not established.
- Safe use of `actual` depends on confirming it is always a type summary, not raw input.
