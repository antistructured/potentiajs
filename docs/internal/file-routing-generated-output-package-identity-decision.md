## Generated output policy

### Default output path

Default generated output path:

```txt
.potentia/routes.generated.js
```

Rationale:

- keeps generated files isolated
- avoids mixing generated code into hand-authored source trees
- matches current internal writer/script behavior
- can be ignored by default
- works for explicit app imports such as `import routes from './.potentia/routes.generated.js'`

### Generated header

Future public generated output should start with:

```js
// Generated by PotentiaJS. Do not edit by hand.
```

The header should also eventually include enough context to debug safely, such as root/output/package name, without embedding absolute local paths by default.

### Runtime behavior

Generated output must:

- import route modules explicitly
- import `createRoutes` / `mount` from the configured package name
- export explicit `createRoutes(...)`
- contain no filesystem scanning code
- contain no Bun-only runtime filesystem APIs
- not execute route handlers during generation
- not mutate global state
- be inspectable JavaScript

Generated output must not:

- scan the filesystem during production request handling
- boot a server
- register routes globally
- hide collisions
- embed absolute local paths by default

## Package identity policy

### Default package name

Default generated import package:

```txt
@potentiajs/core
```

Reason: this is the current package identity and GitHub release install shape.

### Override

Keep `packageName` override in the public API:

```js
await generateFileRoutes({
  rootDir: 'routes',
  outputFile: '.potentia/routes.generated.js',
  packageName: '@potentiajs/core'
});
```

Valid use cases:

- local/self-import tests
- monorepo package aliases
- forks
- future package rename or split, if explicitly chosen later

### Stale package-name prevention

Future implementation should avoid stale package identity by:

- deriving default package name from package metadata where practical
- keeping a single package-name constant in file-routing code/tests if derivation is not practical
- asserting generated output imports `@potentiajs/core` in package identity checks
- using explicit `packageName` overrides in fixture tests rather than old hard-coded names
- documenting package-name assumptions in the design report

## Commit / ignore policy

`.potentia/` remains ignored by default.

Generated files:

- should not be shipped in the npm package
- may be generated in temp/test fixtures
- may be committed by an application author if they intentionally want generated code in source control, but Potentia should not require this
- should be safe to delete/regenerate

Package `files` should not include:

- `.potentia/`
- `src/dev/file-routing/` until the public subpath is intentionally promoted
- internal test fixtures
- `docs/internal/`

## Determinism policy

Generated output must be deterministic for the same input tree and options:

- stable file traversal ordering
- stable import names
- stable grouping/mount order
- stable diagnostics ordering
- stable output formatting

## Source path policy

Default generated output should use relative paths for imports/source metadata.

Absolute paths are deferred because they leak local machine details and make generated files non-portable.

## Decision

- default output: `.potentia/routes.generated.js`
- default package import: `@potentiajs/core`
- package override: allowed
- generated warning header: required in future public implementation
- deterministic output: required
- runtime scanning: forbidden
- generated files: ignored by default, not shipped by package

## Deferred

- source map comments
- generated manifest file output
- route-source metadata implementation
- commit-vs-ignore public recommendation beyond "ignored by default"
- package-name derivation implementation
